## HIGH

### H1. `ai_mode = 'off'` is only enforced on 2 of ~9 AI endpoints (privacy promise broken)
`userPrefs.js:5-8` promises "a stale or hostile client must not be able to run AI for a user who turned it off," but only smart-capture (`routes/ai.js:80`) and weekly-review (`:732`) call `getAiMode`. Unenforced: process-inbox (`:203`), daily-priorities (`:271`), plan-day (`:403`), import-notes (`:556`), find-duplicates (`:644`), task analyze (`routes/tasks.js:148`), project breakdown (`routes/projects.js:70`), list extract-url (`routes/customLists.js:10`). An off-mode user's full inbox (titles + notes) can be shipped to Groq/OpenAI by any authenticated request.
**Fix:** small middleware (`requireAiMode`) applied to every AI-invoking route.

### H2. find-duplicates: hallucinated model IDs drive real deletion
`aiSchema.js:137-151` coerces the model-returned task `id` but never validates it against the prompted task set; `apply-duplicates` (`routes/ai.js:668-681`) deletes whatever `taskIds` arrive. Model transposes an ID → user confirms deleting "duplicate" → a different task of theirs is permanently deleted. Same unvalidated-ID class in `validateWeeklyReview` (`aiSchema.js:153-167`), milder consequences.
**Fix:** filter model IDs to the input set in the validator; render group titles from DB rows, not model output.

### H3. Free-tier capacity gates bypassable two ways (monetization)
- **Habit reactivation:** `PUT /habits/:id` (`habits.js:256`) sets `active` with no `assertWithinLimit`; the cap counts only `active=true` (`billing.js:41`). Archive → create → reactivate = unlimited habits.
- **Import commit:** `import.js:229-243, 274-284` creates projects/habits with no gate at all; a hand-edited export JSON blows past every limit.
**Fix:** gate reactivation transitions and import-commit creations.

### H4. Cross-midnight calendar events invert busy ranges → planner double-books (live-repro'd)
`scheduling.js` `collectBusyRanges`: start/end projected to minutes-of-day independently, so an event ending on a later day yields `[900, 720]`, breaking `freeRangesFrom`'s cursor and emitting overlapping free windows — verified repro placed a 16:00 block *inside* a 15:00→next-day event. The event's next-day tail is also invisible to tomorrow's plan (timed events get a single `due_date`; only all-day events are multi-day-expanded).
**Fix:** clamp per-day (start<dateStr → 0, end>dateStr → 1440) using user-tz date keys; expand timed multi-day events per day.

### H5. GCal fetch window still built in server-local UTC
`googleCalendar.js:157-158`: `timeMin = new Date(startDate+'T00:00:00').toISOString()` — same class as the bug fixed in `7ca7ed7`, one layer up. On UTC Railway a US-Pacific user's "today" window is 17:00-yesterday→16:59-today: a 17:00–18:00 meeting today is never fetched and the planner books over it. Latent for CET users (clipped span is 00:00–02:00 local).
**Fix:** widen the query window ±1 day (cheap; the existing `due_date` filter already prunes) or compute true user-tz day bounds.

### H6. Replan-after-apply silently destroys the applied plan (×2)
`routes/ai.js:449-454` upsert clears `applied_at` at **proposal** time. Applied plan at 9:00 → click "Plan my day" at 14:00 → "Not today" → blocks are still live but `plan.applied=false`: DayBrief progress + the entire DayShutdown card vanish for the day, and `daysPlanned` (retention metric) drops. Variants: applying plan #2 clears `is_daily_focus` on plan #1's unfinished blocks (`:478-481`), orphaning them from progress/shutdown/deferral forever; and any generated-then-canceled proposal burns a free-tier monthly plan credit (`billing.js` counts rows, not applies) and kills the morning CTA banner (`DayBrief.jsx:55` requires `!brief.plan`).
**Fix:** clear/replace `applied_at` only in apply-plan; count only applied plans toward the monthly gate; on apply, only clear `is_daily_focus` for tasks not still scheduled.

### H7. HabitCalendar computes "today" in UTC; server now uses user-local today
`HabitCalendar.jsx:10-14` — the header comment's premise ("matching the server, today is computed in UTC") went stale when `req.today` became timezone-aware. Prague users at 00:00–02:00 local: today's cell renders "future"/disabled (can't log; on the 1st, can't even navigate to the current month). US users in the evening: popover pre-selects local *tomorrow* and lets them log it. Same UTC-today pattern in `Habits.jsx:47-50` (card state goes stale after popover logging), `Habits.jsx:365` (heatmap), `RestDaysModal.jsx:5`, `CompletedTasks.jsx:116`.
**Fix:** one local `todayStr()` helper (local date, not `toISOString`) used everywhere.
